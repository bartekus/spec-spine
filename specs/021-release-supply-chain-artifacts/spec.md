# 021: Release supply-chain artifacts (per-target SBOM and build provenance)

## 1. Purpose

The release pipeline (spec 007, `release.yml`) builds five per-triple archives
and attaches them to a GitHub Release. A binary with no bill of materials forces
a downstream consumer to reverse-engineer what is inside it, and an archive with
no provenance cannot be tied back to the workflow run that produced it. This spec
closes both gaps the way OAP does (spec 117): a CycloneDX SBOM and a build
provenance attestation per shipped artifact.

The architecture's open question Q7 deferred SBOMs for v1 ("low value/time
ratio"); this spec is the deliberate decision to add them now, on request.

## 2. Territory

This spec establishes no new files. It is an additive extender of spec 007:

- **`.github/workflows/release.yml`** (additive): new steps in the `build` job
  that generate, validate, and attest per-target supply-chain artifacts, and the
  job permissions those steps require.
- **`docs/releasing.md`** (additive): a section documenting the artifacts and
  how to verify them.

007 remains the owner of both files; 021 only adds.

## 3. Behavior

For each of the five release triples, the `build` job MUST, after producing the
archive and its `.sha256` sidecar:

1. **Generate a CycloneDX SBOM** with `anchore/sbom-action` (syft), format
   `cyclonedx-json`, scanning the repository so the Rust cataloger reads the
   committed `Cargo.lock`. The SBOM file is named to match its archive:
   `spec-spine-<tag>-<triple>.cdx.json`.
2. **Fail closed on an empty SBOM.** If the generated SBOM has zero components,
   the job MUST fail rather than ship a vacuous SBOM (the OAP AC-6 guard). This
   catches a cataloger regression (a syft version that returns no Rust
   components) instead of silently shipping nothing.
3. **Attest build provenance** over the archive with
   `actions/attest-build-provenance`, which records a SLSA provenance statement
   in GitHub's attestation store. The job carries `id-token: write` and
   `attestations: write` for this; no long-lived token is stored.

The `publish` job attaches every `*.cdx.json` to the GitHub Release as a
separate asset alongside the archives (it already globs the build artifacts).
A consumer verifies an archive with:

```sh
gh attestation verify spec-spine-<tag>-<triple>.tar.gz --repo bartekus/spec-spine
```

The SBOM and attestation are **deterministic-input metadata**: the dependency
list comes from `Cargo.lock`, which is committed and pinned, matching the binary
the determinism gate proves byte-identical across triples.

## 4. Out of scope

- crates.io / npm / PyPI publishing semantics (spec 007 / 008). Those channels
  are untouched; this is binary-release metadata only.
- A repo-wide dependency-audit gate (`cargo-deny` and friends). That is a
  separate supply-chain concern (OAP spec 116) and is not adopted here.
- Attaching the raw attestation bundle as a release asset. Provenance is recorded
  in GitHub's attestation store and verified with `gh attestation verify`; a
  bundle asset for offline verification can be a later additive refinement.
- Committing any SBOM. SBOMs are release-time artifacts, assembled per tag and
  never committed (the same posture as the npm/PyPI generated packages).
